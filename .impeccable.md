# Impeccable Design Context — easy-review (`er`)

## Design Context

### Users
Developers who use AI coding assistants (Claude Code, Copilot, Cursor). They generate code faster than they can review it and need a tool that keeps pace. They live in the terminal, use vim keybindings instinctively, and value speed over polish. Context: focused review sessions where every second of friction compounds.

### Brand Personality
**Sharp, Focused, Fast.** No decoration. No hand-holding. Every pixel earns its place. The tool should feel like an extension of the developer's workflow — invisible when working, unmistakable when needed. Terse naming (`er`), keyboard-only interaction, real-time file watching all reinforce this identity.

### Aesthetic Direction
- **Visual tone**: Minimal, professional, high-contrast dark (with theme system planned)
- **Current theme**: "Ocean Depth" — cool blue-undertone dark palette. Backgrounds use blue-purple tints; accents are vivid (blue, cyan, green, orange)
- **Theme system**: Introducing multiple themes (dark, light, and variants). All colors should flow through a theme abstraction — no hardcoded values in rendering code. This is an opportunity to refresh the full palette
- **Anti-references**: Avoid flashy/decorative TUIs, colorful-for-colorful's-sake palettes, or anything that feels like a demo rather than a daily tool
- **Syntax highlighting**: `base16-ocean.dark` via Syntect (will need theme variants)

### Design Principles

1. **Information density over whitespace.** Terminal real estate is precious. Every row should carry meaning. Padding is intentional, never decorative.

2. **Semantic color, not decorative color.** Colors communicate state: green=added, red=removed, yellow=warning, cyan=comment, purple=selection. If a color doesn't convey meaning, remove it.

3. **Contrast creates hierarchy.** Three text levels (bright/normal/dim) plus accent colors. No gradients, no shadows — just contrast ratios that work at 2am.

4. **Theme-ready architecture.** All colors route through a theme layer. Components never reference specific hex values — they reference semantic tokens (`bg`, `surface`, `text.primary`, `accent.action`). New themes should work by swapping the token map, not editing components.

5. **Speed is a feature.** Viewport-based rendering, cached highlighting, debounced updates. Design decisions should never trade performance for aesthetics. If it's not instant, it's wrong.

### Color Token Structure (Theme System)

When implementing themes, colors should be organized as semantic tokens:

```
Background layer:    bg, surface, panel, border
Text layer:          text.primary, text.bright, text.dim, text.muted
Diff layer:          diff.add.bg, diff.add.text, diff.del.bg, diff.del.text, diff.hunk.bg
Accent layer:        accent.action (blue), accent.info (cyan), accent.success (green),
                     accent.warning (yellow), accent.error (red), accent.selection (purple),
                     accent.emphasis (orange)
Interactive layer:   cursor.bg, focus.bg, comment.bg, finding.bg
Status layer:        stale, watched.text, watched.bg, unmerged, relocated, lost
```

### Component Inventory

- **Layout**: Two-column (file tree + diff) with optional third column (AI panel)
- **Status bars**: Top (tabs, branch, mode selector) + bottom (keybindings, input)
- **File tree**: Left panel with risk dots, status colors, reviewed checkmarks
- **Diff view**: Right panel with viewport rendering, syntax highlighting, inline comments
- **Side panel**: Tabbed (File/AI/PR) with findings, comments, metadata
- **Overlays**: Centered modals for worktree picker, directory browser, filter, settings
- **Inline banners**: AI findings (warm bg) and comments (cool bg) embedded in diff
- **Markers**: Unicode symbols for state (triangle, dot, check, diamond, circle)
