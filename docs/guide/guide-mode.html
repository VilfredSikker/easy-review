Core Concepts

Guide Mode

A diff lists files in whatever order git emits them — usually alphabetical, which is almost never the order that makes a change easy to understand. Guide mode fixes that. It is an AI-generated walkthrough that groups the changed files into ordered pillars — foundational changes first, then by importance — so you read the change as a story instead of an alphabetised pile. It works in both the terminal UI (where it is a Tour tab) and the desktop app (where it is the Guide toggle), and both read the same file.

◆

er never runs AI itself. Guide mode renders a walkthrough that an AI tool you run writes to disk as .er/tour.json. Generating one is a single skill call (or one button in the desktop app); see Generating a guide below.

What a guide is made of

A guide is a small JSON sidecar (tour.json) describing a self-contained reading plan for the current diff. It never reorders or overwrites your other review data — it keeps its own grouping and only references the files (and optionally AI findings) already in the diff. It has three parts:

  • A title and overview — a one-line title and a short markdown intro for the whole change.
  • Pillars — ordered groups of related files. Each pillar has a title, a markdown description (the “why” for that group), and the list of files it covers in intended reading order.
  • Per-file reasons — within a pillar, each file can carry a short reason it belongs there and links to any AI review findings that touch it.

How pillars are ordered

Pillars are sorted so you always read the groundwork before the things built on top of it. The ordering is:

  1. Explicit order — a lower order value sorts earlier.
  2. Foundation first — at equal order, pillars marked foundation (shown with a ◆ marker) come before the rest. These are the schema changes, new types, and core plumbing that the rest of the diff depends on.
  3. Importance — a 0–100 reviewer-attention weight breaks remaining ties (higher first) and doubles as a badge.
  4. Title — alphabetical, as a final tie-break.

Any changed file the guide does not place into a pillar is collected into a trailing “Other changes” pillar, so the walkthrough always covers the whole diff — nothing silently disappears. Guide mode only applies to the branch diff; that is the scope a walkthrough is generated against.

Generating a guide

There are two ways to produce a tour, and they share the same prompt and output format.

From the desktop app desktop

Open the AI action palette and choose Generate tour (it reads Regenerate tour once one exists). This spawns the er-tour agent as a background task; when it finishes, the tour is written straight into the branch's shared storage and the Guide toggle appears automatically — no manual step. This works because the desktop spawns the agent as a normal child process, so it can write shared storage directly.

With the /er-tour skill both

Run the skill from your AI assistant in the repo you are reviewing:

/er-tour

It analyses the current diff, groups the files into pillars, and writes a single .er/tour.json in the repo. Skills run inside your AI assistant's sandbox, which can write the repo's .er/ directory but not the shared storage the apps read from — so a hand-run tour lands in .er/ and needs one more step to go live: import it.

Importing .er/ files into shared storage both

er itself runs outside that sandbox, so it can move skill-written sidecars into shared storage for you. Whenever .er/ holds review files that are newer than — or missing from — shared storage, the apps offer to import them:

  • desktop an Import local review files action appears in the AI action palette.
  • terminal press I to import.

Import copies tour.json (and any other skill output — review.json, summary.md, checklist.json, and the rest) into the bucket the apps read from, and the Tour tab / Guide toggle appears on the next refresh. If a file would overwrite an existing copy in shared storage, er asks you to confirm first and tells you which copy is newer — your .er/ version or the one already in storage — so you never silently clobber fresher data.

Guide mode in the terminal terminal

In the TUI, the guide is a numbered Tour tab that only shows when a tour exists for the branch (and the feature is enabled — see Enabling & disabling). Because tab numbers are assigned dynamically, the Tour tab takes whatever number is next in your enabled-modes list. Switch to it with that number key, the same way you switch any diff mode.

The view is two panels:

  • Left — the pillar list. Every pillar with its title, a ◆ marker if it is foundational, and a progress badge: a green ✓ when all its files are reviewed, otherwise reviewed / total. Files are nested under each pillar with a ✓ beside the ones you have reviewed.
  • Right — the diff, reordered. The branch diff laid out in pillar order rather than file order, with a sticky header showing the current pillar (TOUR · pillar N/M) and its description above the changes.

Navigating a guide

KeyAction
j / kPrevious / next pillar (jumps to the pillar's first file).
n / NNext / previous file, crossing pillar boundaries.
↑ / ↓Move line by line through the diff.
u / dPage up / down; the selected pillar re-syncs to what is on screen.
h / lScroll the diff horizontally for long lines.
SpaceToggle the current file as reviewed.
bBulk-review: mark every file in the selected pillar as reviewed.

Reviewed state is shared with the rest of er — files you mark reviewed in Guide mode stay reviewed in Branch mode and vice versa. If no tour exists yet, the panel shows “No tour available — run /er-tour to generate one.”

Guide mode in the desktop desktop

In the desktop app the guide is a Guide / Diff toggle in the top-right of the diff header. The toggle only appears once a tour exists for the branch; click Guide to enter the walkthrough and Diff to return to the normal branch diff.

Guide mode turns the diff into a Split View:

  • A pillar rail on the left. A sticky lane that pins each pillar's header to the top of the viewport while its files scroll past. Each pillar shows its title, a foundation badge (◆), its description, a reviewed / total count (or a green Reviewed badge when complete), a Review all / Unreview all button, and the file list with a ✓ and ± line counts per file.
  • The diff, reordered by pillar. Files are laid out in the pillar sequence rather than alphabetically, with the “Other changes” group last.
  • Auto-collapse on review. When you mark a file reviewed in Guide mode it collapses to a compact row, keeping your attention on what is left. Already-reviewed files start collapsed; you can always expand one manually.

Staying fresh

A guide is generated against a specific snapshot of the diff. Each tour.json records the hash of the diff it was built from; when the underlying diff changes — a watched file is saved, something is staged or committed — er can tell the guide no longer matches. The walkthrough still renders (files that have since vanished from the diff are simply dropped), but the grouping may be out of date. Regenerate it — re-run /er-tour or hit Regenerate tour — to bring the pillars back in line with the current change.

Enabling & disabling

Guide mode is gated behind the Tour feature flag, which is on by default. You can toggle it from the settings overlay (S in the terminal, the settings screen in the desktop) or in configuration: 

[features]
view_tour = true   # "Show AI guided tour mode (when a tour exists)"

Turning it off hides the Tour tab and the Guide toggle regardless of whether a tour.json is present. With it on, the tab/toggle still only appears when a guide actually exists for the branch — so it never adds clutter to a diff you have not generated a walkthrough for.

●

Guide mode pairs well with AI Review: pillars can link directly to review findings, so a walkthrough doubles as a route through the riskiest parts of the change. Generate the review first, then the tour, to get findings threaded into the pillars.
