Core Concepts

Comments & Questions

er keeps two separate streams of inline notes: questions you keep to yourself (and hand to your AI), and GitHub comments that sync to a pull request. They look different, use different keys, and never get confused with one another — so a private thought can't accidentally end up on a PR.

Why two kinds?

Questions ●GitHub comments ●
ColorYellowCyan
Keysq line · Q hunkc line · C hunk
AudienceYou and your AI toolsYour PR reviewers
Stored inquestions.jsongithub-comments.json
Replies?No (use AI to answer)Yes — single-level threads
Synced upstream?Never — stays localYes, two-way via gh

Questions are your scratchpad: “why is this here?”, “does this handle the empty case?”. The /er-questions skill reads them, answers, and updates the file in place. GitHub comments are the things you actually want your team to see on the PR.

Leaving a note

Navigate to a line (with the arrow keys) or sit on a hunk, then press the matching key. Lowercase targets the current line; uppercase targets the whole hunk.

KeyAction
qAdd a question on the current line
QAdd a question on the current hunk
cAdd a GitHub comment on the current line
CAdd a GitHub comment on the current hunk

A small input opens at the bottom; type your note and confirm. The note then renders inline, right after its target line in the diff — not bundled at the end of the hunk — so the conversation sits exactly where the code is.

Comment focus mode

To act on existing notes rather than the diff, enter comment focus mode with Tab. Now the arrow keys move between comments instead of lines, and these actions apply to the focused comment:

KeyAction
TabEnter / leave comment focus mode
↓ / ↑Move between comments
rReply to the focused comment (GitHub comments only)
dDelete the focused comment
RToggle resolved

Replies are single-level

GitHub comments support a reply thread: a parent comment plus any number of replies. Replies cannot themselves have replies — the model is intentionally flat (parent + children, no recursive nesting) to keep it simple and to match how GitHub PR review threads behave. Questions do not have replies; the way you “answer” a question is to run /er-questions and let the AI respond.

Comments follow the code (and warn when stale)

Each note stores the content of the line it was attached to. When the diff changes, er tries to re-anchor the note to that same line even if it moved. If the target line disappears entirely, the note is marked stale: it renders dimmed with a warning indicator so you know it may be misplaced and should be revisited. This staleness is tracked per comment, independently of the file-level AI staleness described in AI Review.

Syncing GitHub comments

With the gh CLI configured and a pull request in view, comments sync both directions:

KeyAction
GPull review comments from GitHub into your diff
PPush your local comments to GitHub

Each GitHub comment records its source (local or github), the upstream github_id, the author, and whether it has been synced. Pulling deduplicates on the GitHub id, so you can pull repeatedly without creating copies. The full sync flow lives in GitHub & Pull Requests.

✦

Rule of thumb: use q while you are still figuring out whether something is a problem, and c once you are ready to say it out loud on the PR. /er-publish includes your GitHub comments when it posts the review.
