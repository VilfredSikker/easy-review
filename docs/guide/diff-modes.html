Core Concepts

Diff Modes

A “mode” is simply which set of changes er is showing you. There are six built-in modes plus a pull-request mode. In the terminal you switch with the number keys; in the desktop app you switch from the branch context bar.

The six modes at a glance

KeyModeShows
1BranchEverything that differs between your branch and its detected base branch.
2UnstagedWorking-tree changes you have not staged yet (git diff).
3StagedChanges in the index, ready to commit (git diff --cached).
4HistoryCommit-by-commit view of your branch's history.
5ConflictsFiles with unresolved merge conflicts.
6HiddenHidden / git-ignored files that er can still surface.
◆

Tab numbers are assigned dynamically to the modes that are enabled. If you disable a mode in configuration, the remaining modes renumber so there are no gaps.

Branch both

The default mode and the one you will use most. It answers “what does this branch change?” by diffing against the auto-detected base branch (see base branch detection). This is the right mode for reviewing a feature branch or a pull request's worth of work.

Unstaged both

Your in-progress edits that have not been git add-ed. Pairs naturally with watch mode while an AI assistant is actively editing — you see changes appear as they are written.

Staged both

What is in the index and would go into your next commit. From the terminal you can stage or unstage the current file with s, and in staged mode you can commit and push without leaving er (Ctrl+P). After you commit, er automatically switches to a view of exactly what you just committed (git diff HEAD~1 HEAD) so you can confirm it before moving on.

History both

A commit log for your branch. Selecting a commit loads its full diff (the parser runs git log plus a git diff per commit). Recent commits are cached so scrolling back through history stays fast. Use this to review work commit by commit rather than as one big squashed diff.

Conflicts both

Surfaces files with unresolved merge conflicts so you can find and work through them quickly during a merge or rebase.

Hidden / ignored both

Files git normally hides — ignored paths and the like. This is distinct from the watched files feature (configurable globs for git-ignored paths such as agent sync folders), which you toggle with W. See Watched files.

Pull request mode both

Not a number key — you enter it by opening a PR. From the terminal: 

er --pr 42
er https://github.com/owner/repo/pull/42

In PR mode the right-hand panel shows CI checks, reviewer decisions, and labels, and comment sync targets that pull request. The desktop app has a “PR Diff” toggle and a paste-a-URL modal. PR review can be done against remote refs without checking out the branch. See GitHub & Pull Requests.

Sorting by recency

In any mode, press Shift+R to toggle sorting files by modification time. When an AI tool just touched a handful of files, this floats them to the top of the list so you review the freshest changes first.

Performance with large diffs

er is built to stay fast even on enormous diffs. Three mechanisms do the heavy lifting:

  • Auto-compaction — lock files, generated code, minified assets, and files over ~500 lines are compacted automatically: their hunks are cleared from memory and shown as a single collapsed entry. Press Enter to expand one on demand.
  • Two-phase lazy parsing — diffs above ~5,000 lines get a header-only scan for the file list, and individual files are parsed only when you navigate to them.
  • Viewport rendering — only the rows visible on screen (plus a small buffer) are ever built, so a 5,000-line file renders like a 60-line one.

You rarely have to think about any of this — it just keeps scrolling smooth.
